## 1️⃣ 페르소나 기법(Persona Prompting)이란?

LLM에게 **특정 역할·전문성·성향·목적**을 부여하여 **일관된 출력 스타일과 사고방식**을 유지하게 만드는 프롬프트 설계 기법입니다.

즉,

> “너는 어떤 사람처럼 행동해라. 어떤 방식으로 말하고, 어떤 목적을 가지고 사고해라.”

라고 지시함으로써  
🔥 **정답 품질을 안정화하고**,  
🔥 **문체·논리·전문성·해결방식**을 원하는 방식으로 맞추게 할 수 있습니다.

---

## 2️⃣ 페르소나 기법이 필요한 이유

### ✔ 모델의 ‘기본 성격’은 일반적이고 무난함

→ 전문가처럼 깊게 분석하거나, 특정 스타일로 말하게 하려면 **명확한 역할 정의**가 필요함.

### ✔ 복잡한 업무에서는 사고의 일관성 유지가 매우 중요

예:

- 보안 아키텍처 전문가 페르소나 → 모든 답변에 리스크 고려
    
- RAG 튜닝 전문가 페르소나 → 컨텍스트 전략을 중심으로 사고
    
- 기업 CISO 페르소나 → 정책·위험관리에 초점
    

### ✔ Agentic RAG·Tool-Calling에서 필수

MCP, LangGraph 같은 구조에서는  
‘LLM이 어떤 관점으로 사고하느냐’가 에이전트 행동 품질을 크게 좌우함.

---

## 3️⃣ 페르소나 프롬프트의 기본 구조

### 🧱 **[1] Role — 어떤 사람인가?**

- 직업, 레벨(전문가/멘토/리더 등)
    
- 도메인 전문성
    
- 사고 스타일
    

### 🧱 **[2] Goal — 무엇을 달성하려고 하는가?**

- 답변 목적
    
- 문제 해결 방식
    
- 성과 지향점
    

### 🧱 **[3] Style — 어떻게 말하고 행동하는가?**

- 톤/문체
    
- 단계적 설명 여부
    
- 난이도
    

### 🧱 **[4] Constraints — 반드시 지켜야 하는 제한 조건**

- 금지사항
    
- 포맷
    
- 비유 방식
    
- 근거 포함 여부
    

---

## 4️⃣ 페르소나 프롬프트 템플릿

아래 템플릿은 어떤 작업에도 그대로 적용 가능:

`[ROLE] 당신은 {직업/역할}입니다. {전문성/경력}을 보유하고 있으며 {업무 방식, 사고 스타일}을 가지고 있습니다.  [GOAL] 당신의 목표는 {유저의 목적/문제 해결 목표}입니다.  [STYLE] - 톤 & 스타일: {예: 간결/친절/전문적/엄격/논리적} - 설명 방식: {예: 단계별, 표 포함, 예제 중심} - 우선순위: {예: 정확성 > 속도, 보안 고려 최우선}  [CONSTRAINTS] - {피해야 하는 내용} - {출력 형식} - {추가 요구사항}`

---

## 5️⃣ 실전 예시 (개발·AI·보안 중심)

### 📌 예시 1: RAG 아키텍트 페르소나

`당신은 대규모 AI 시스템을 설계하는 ‘RAG 아키텍처 전문가’입니다. LangChain·LangGraph·Vector DB 최적화·Chunking 설계 경험을 갖고 있습니다.  목표: - 주어진 문제를 해결하는 최적의 RAG 구조를 설계합니다. - 정확성/효율성/비용 균형을 고려합니다.  스타일: - 단계별 reasoning - 다이어그램/테이블 적극 사용 - 실무 중심  제약: - 이론만 말하지 말고 구체적 구조 제시 - 코드 예시가 있을 경우 Python/LangChain 중심`

---

### 📌 예시 2: 상용 서비스 보안 아키텍트 페르소나

`당신은 15년 경력의 보안 아키텍트이자 CISO입니다. 데이터 암호화, 개인정보보호, E2EE, API 보안에 정통합니다.  목표: - 사용자가 제시한 시스템의 보안 취약점 식별 - 개선 아키텍처 제안 (암호화, 인증, 토큰, 키 관리 포함)  스타일: - 실무 중심, 모호한 표현 금지 - "리스크 → 영향 → 해결 방안" 구조로 답변  제약: - 과도하게 이론적인 언급 배제 - 실제 기업 환경에서 적용 가능한 아키텍처만 제안`

---

### 📌 예시 3: 시니어 Spring Boot 백엔드 엔지니어 페르소나

`당신은 10년 경력의 Spring Boot 시니어 백엔드 엔지니어입니다. 필터, 인터셉터, AOP, ResponseBodyAdvice, AES 암호화, JWT 인증 등을 전문적으로 다룹니다.  목표: - 사용자의 문제를 가장 간단하고 안정적으로 해결하는 백엔드 코드를 제시 - 레거시 구조가 있을 경우 개선 포인트도 함께 제안  스타일: - 불필요한 boilerplate 최소화 - 코드와 설명을 함께 제공 - 문제 원인을 먼저 진단 후 해결책 제시`

---

## 6️⃣ 페르소나 기법이 잘 먹히는 작업들

### ✔ 복잡한 기술 컨설팅

- RAG 아키텍처 구성
    
- Kubernetes, Docker Compose 비교
    
- Spring Security, AES key/IV 처리
    

### ✔ 특정 스타일의 답변이 필요할 때

- 논리적 글쓰기
    
- 기술 블로그 스타일
    
- 초보자용 단계별 튜토리얼
    

### ✔ LLM 에이전트 설계, MCP, Tool-Calling

페르소나가 곧 **"에이전트의 성격 = 행동 정책"**이 됨.

---

## 7️⃣ 페르소나 프롬프트 작성 꿀팁

### ✔ (1) 직업만 주면 안된다 → "사고방식"까지 정의해야 효과가 큼

예:  
❌ “너는 백엔드 개발자야.”  
⭕ “너는 문제 원인을 로그 레벨에서 먼저 찾고, 예외 흐름을 반드시 분석하는 백엔드 개발자.”

### ✔ (2) 페르소나는 너무 길 필요 없다

핵심은 _일관성_, _행동 정책_, _우선순위_만 명확히 하는 것.

### ✔ (3) 질문마다 페르소나를 재정의할 필요 없음

지속되는 세션에서는 **한 번 정의한 페르소나 + 추가 규칙**만 붙이면 됨.

### ✔ (4) “금지사항”을 넣으면 정확도 폭발

예:

- “추상적 설명 금지”
    
- “코드만 단독으로 출력 금지”
    
- “테이블 없이 설명 금지”
    

---

## 8️⃣ 마지막으로, 페르소나 기법의 핵심 요약

|항목|설명|
|---|---|
|**역할(Role)**|어떤 전문가인가?|
|**목표(Goal)**|무엇을 달성하려 하는가?|
|**스타일(Style)**|말투·설명 방식·논리 구조|
|**제약(Constraints)**|꼭 지켜야 할 규칙·형식|

페르소나 기법은 LLM이  
**“내가 누구처럼 생각해야 하는지”**  
를 정교히 이해시키는 핵심 기술입니다.
